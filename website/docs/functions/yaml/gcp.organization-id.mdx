<Intro>
The `!gcp.organization_id` YAML function retrieves the GCP organization ID for the current project
by walking the project's resource ancestry using the Cloud Resource Manager API.
</Intro>

## Usage

The `!gcp.organization_id` function takes no parameters:

```yaml
  # Get the GCP organization ID
  org_id: !gcp.organization_id
```

## Arguments

This function takes no arguments. It uses GCP credentials from the environment or the
Atmos authentication context if configured.

## How It Works

When processing the `!gcp.organization_id` YAML function, Atmos:

1. **Resolves the Project ID** - Determines the active GCP project using the same priority
   as `!gcp.project_id` (auth context → environment variables).

2. **Calls Cloud Resource Manager** - Calls the `projects.getAncestry` API to walk the project's
   resource hierarchy.

3. **Finds the Organization Ancestor** - Scans the ancestry list for an ancestor of type
   `organization` and returns its numeric ID (e.g., `"123456789"`).

:::note Organization Membership Required
The project must be a member of a Google Cloud Organization. Personal projects and standalone
projects without an organization ancestor will return an error.
:::

:::note Required IAM Permission
The caller must have `resourcemanager.projects.getAncestry` permission (included in
`roles/viewer` or `roles/browser`).
:::

## Authentication

### Atmos Auth Integration

When using [Atmos Authentication](/cli/commands/auth/usage), the function automatically uses credentials
from the active GCP identity. This enables seamless integration with Workload Identity Federation,
service account impersonation, and other authentication methods configured in your `atmos.yaml`.

The credential resolution order is:

1. **Atmos Auth Context** (highest priority) - If a GCP identity is active, Atmos uses the
   access token or credentials file from that identity.
2. **Credentials File** - If `credentials_file` is set in the GCP auth context, uses that file.
3. **Application Default Credentials** (fallback) - Falls back to ADC if no explicit credentials
   are provided.

## Caching

The `!gcp.organization_id` function uses its own cache, separate from the identity functions
(`!gcp.project_id`, `!gcp.service_account_email`, etc.). Results are cached in memory for the duration of
the CLI invocation:

- Multiple uses of `!gcp.organization_id` in the same command only make one API call
- Cache is keyed by project ID, so different projects get separate cache entries
- Each new CLI command starts with a fresh cache

Atmos supports type-aware merging of YAML functions and concrete values, allowing them to coexist in the inheritance chain without type conflicts.
See the full explanation: [YAML Function Merging](/reference/yaml-function-merging)
:::

### Basic Usage

<File title="stack.yaml">
```yaml

  terraform:
    my-component:
      vars:
        # Inject the GCP organization ID into Terraform variables
        org_id: !gcp.organization_id
```
</File>

### Organization-Level Policies

<File title="stack.yaml">
```yaml

        organization_id: !gcp.organization_id

</File>

<File title="stack.yaml">

    org-policy:

        org_id: !gcp.organization_id  # Uses cached value

```
</File>

## Error Handling

If the function fails to retrieve the GCP organization info, Atmos will log an error and exit.

Common error scenarios:
- **No project ID configured** - Set `project_id` in GCP auth context or `GOOGLE_CLOUD_PROJECT` env var
- **Project not in an organization** - Personal or standalone projects have no organization ancestor
- **Missing permissions** - The caller lacks `resourcemanager.projects.getAncestry` permission
- **Network connectivity issues** - Cannot reach the Cloud Resource Manager endpoint

## Considerations

- **Requires organization membership** - Projects not part of a GCP Organization will fail
- **Requires `resourcemanager.projects.getAncestry` permission** - Typically available via
  `roles/viewer` or `roles/browser`; check with your GCP admin if permission is missing
- **Network dependency** - Requires connectivity to the Cloud Resource Manager API
- **Separate cache** - Uses a separate cache from the identity-based functions since it calls
  a different GCP API
- **Performance** - Results are cached per CLI invocation, so there is minimal overhead when
  used multiple times
